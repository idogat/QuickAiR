## Document Structure Contract

The document is divided into the following heading hierarchy. Heading text is the anchor target for Phase 2 and Phase 3 cross-references — heading text MUST NOT change after the document is published.

```
# DFIR Artifact Coverage Matrix                      (H1 — document title, one only)
## Legend                                            (H2 — tier definitions and notation key)
## Tier Model                                        (H2 — PS 2.0 / PS 3+ / PS 5.1 description)
## Capability Probe                                  (H2 — how tiers are detected at runtime)
## Processes                                         (H2 — collector section)
## Network — TCP Connections                         (H2 — collector section)
## Network — DNS Cache                               (H2 — collector section)
## DLLs                                             (H2 — collector section)
## Users                                            (H2 — collector section)
## Toolkit-Wide Coverage Gaps                        (H2 — what is never collected at any tier)
```

Each H2 collector section contains three sub-components:

1. **Fallback chain table** — which method at which tier, with exact `source` field string values
2. **Field fidelity table** — which fields are populated, null, or degraded at each tier
3. **Coverage gaps list** — explicit named gaps for that artifact (bulleted list)

Source: CONTEXT.md Claude's Discretion (per-collector sub-tables decision), RESEARCH.md Architecture Patterns.

## Typography

Heading and text conventions for the markdown document. All sizing is relative (rendered by GitHub and local editors).

| Role | Markdown Level | Weight | Usage |
|------|---------------|--------|-------|
| Document title | H1 (`#`) | Bold (rendered by renderer) | One occurrence — document title only |
| Section heading | H2 (`##`) | Bold | Top-level sections: Legend, Tier Model, collector names, Gaps |
| Sub-heading | H3 (`###`) | Bold | Sub-components within a collector section if needed (e.g., "Fallback Chain", "Field Fidelity") |
| Body / cell content | Plain text | Regular (400) | Table cells, list items, prose paragraphs |

Line height: governed by GitHub/editor renderer defaults — not specified.

Code formatting rules:
- All PowerShell class names, method names, CIM class names, field names, and source string values MUST appear in backtick code spans: e.g., `Get-CimInstance`, `MSFT_NetTCPConnection`, `cim`, `dotnet_fallback`
- All file paths and parameter names MUST appear in backtick code spans: e.g., `Modules/Collectors/Network.psm1`, `-TargetCapabilities`

Source: CLAUDE.md Rules (all datetimes UTC, accuracy requirement); CLAUDE.md Conventions (naming patterns).

## Notation Contract

These notation standards replace a color/icon system for a markdown document. All contributors MUST use these exact strings in table cells — no variations, no synonyms.

### Tier Labels (column headers — exact text required)

| Column header text | Meaning |
|-------------------|---------|
| `PS 5.1 / Full CIM` | PSVersion >= 3 AND HasNetTCPIP=true AND HasDNSClient=true |
| `PS 3+ / Partial CIM` | PSVersion >= 3 AND (HasNetTCPIP=false OR HasDNSClient=false) |
| `PS 2.0 / Legacy` | PSVersion < 3 |

Source: CONTEXT.md D-01, RESEARCH.md Tier Model table.

### Cell State Notation

| State | Notation | Usage |
|-------|----------|-------|
| Field fully populated, expected value | `populated` | Field present and correct at this tier |
| Field always null — confirmed in source | `null (all tiers)` | Field initialized to $null and never set at any tier |
| Field null at this tier only | `null` | Field absent at this specific tier |
| Field degraded — reduced precision | `derived — {reason}` | Value present but less accurate than preferred tier |
| Field uses different source at this tier | `{source} — {note}` | E.g., `WMI CIM_DataFile` instead of `.NET FileVersionInfo` |
| Field uses string instead of integer | `string (regex parse)` | E.g., DNS Type field at ipconfig tiers |
| Field uses PS2 binary-only status | `PS2_SIGNED / PS2_UNSUPPORTED` | Signature.Status at PS 2.0 WMI tier |
| Collection method at tier | backtick code span | E.g., `Get-CimInstance MSFT_NetTCPConnection` |
| Source field string value | backtick code span | E.g., `cim`, `netstat_fallback`, `wmi+dotnet_fallback` |

### Coverage Gap Notation

Coverage gaps in the gaps column and the Toolkit-Wide Coverage Gaps section MUST use this format:

```
- **{Gap name}**: {one-sentence description of what is missing and why}
```

Example:
```
- **Protected processes skipped**: `lsass`, `csrss`, `smss`, `wininit`, `System`, `Idle` are never
  enumerated at any tier — hardcoded exclusion list in DLLs.psm1.
```

Source: CONTEXT.md D-13 through D-17, RESEARCH.md Collector-by-Collector Findings.

## Table Column Layout Contract

Per CONTEXT.md D-02, every artifact table MUST use this exact column order:

```
| Artifact | Data Fields | PS 5.1 (Full CIM) | PS 3+ (Partial CIM) | PS 2.0 (Legacy) | Fidelity Notes | Coverage Gaps |
```

For the field fidelity sub-tables within each collector section, columns are:

```
| Field | PS 5.1 / Full CIM | PS 3+ / Partial CIM | PS 2.0 / Legacy |
```

For the fallback chain sub-tables:

```
| Tier | Condition | Collection Method | source field value |
```

Source: CONTEXT.md D-02, RESEARCH.md Architecture Patterns (Pattern: Per-Collector Sub-Tables).
