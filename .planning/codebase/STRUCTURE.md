## Directory Purposes

**Root Level:**
- Purpose: Entry points (Collector.ps1, Executor.ps1, QuickAiRLaunch.ps1, Report.html) and project documentation
- Contains: Three independent PowerShell orchestrators, HTML GUI, helper scripts
- Key files: `Collector.ps1` (primary collection), `Executor.ps1` (remote execution), `QuickAiRLaunch.ps1` (job manager)

**Modules/Core/:**
- Purpose: Shared utility modules (lowest architectural layer)
- Contains: Connection management, output/logging, date/time formatting, tools manifest
- Key files: `Connection.psm1` (WinRM session creation, capability probing), `Output.psm1` (JSON serialization, logging)
- Pattern: Always imported first; no dependencies on other modules

**Modules/Collectors/:**
- Purpose: Pluggable data collection modules (middle layer)
- Contains: Process list, network connections, DNS cache, DLL inventory, user sessions
- Key files: `Processes.psm1` (largest, ~800 lines), `Network.psm1` (complex 3-tier fallback)
- Pattern: Auto-discovered by Collector.ps1; each exports `Invoke-Collector` function
- Dependency: Imports Core\DateTime.psm1; receives PSSession from Collector.ps1

**Modules/Executors/:**
- Purpose: Three independent remote execution methods (chained in priority order)
- Contains: WinRM executor (preferred), SMB+WMI executor (fallback), WMI executor (legacy)
- Key files: `WinRM.psm1` (uses PSSession), `SMBWMI.psm1` (file transfer + WMI), `WMI.psm1` (direct execution)
- Pattern: Each exports `Invoke-Executor` function; receives same parameters, returns state transitions
- Dependency: Imports Core\Output.psm1 for logging

**Modules/Launcher/:**
- Purpose: Job queue management and concurrency control
- Contains: Thread-safe job queue, named pipe listener for job batch delivery
- Key files: `JobQueue.psm1` (enforces status transitions), `PipeListener.psm1` (bridge file support)
- Pattern: Imported by QuickAiRLaunch.ps1; maintains singleton queue in memory

**ReportStages/:**
- Purpose: HTML and JavaScript components assembled into single-file Report.html
- Contains: Styling, shell HTML, tabs (Fleet, Processes, Network, DNS, Users, Manifest, Execute), core utilities
- Key files: `03_core.js` (35KB, data loading and state), `09b_execute.js` (26KB, job UI and launching)
- Pattern: Concatenated in order at build time; fully offline, no external dependencies
- Output: `Report.html` (single file, ~200KB when assembled)

**DFIROutput/:**
- Purpose: Output directory for Collector.ps1 results
- Contains: One subdirectory per target hostname; JSON files with collected artifacts
- Key files: `<hostname>\<hostname>_<timestamp>.json` (manifest + processes + network + dns)
- Pattern: Created by Collector.ps1 at runtime; analyst customizes via -OutputPath parameter

**ExecutionResults/:**
- Purpose: Output directory for Executor.ps1 results
- Contains: JSON files with execution metadata and state transitions
- Key files: `<hostname>_execution_<timestamp>.json` (ExecutionId, method, states array, final state, PID)
- Pattern: Created by Executor.ps1 at runtime

**Tests/:**
- Purpose: Automated validation suite
- Contains: 10 test scripts (T01-T10) covering all collection types and execution methods
- Key files: `TestSuite.ps1` (runner), individual T*.ps1 files
- Pattern: Run against collected JSON files to validate completeness and integrity

## Key File Locations

**Entry Points:**

- `C:\DFIRLab\repo\Collector.ps1`: Main collection orchestrator; imports Core modules, auto-discovers Collectors, invokes capability probe, manages target loop
- `C:\DFIRLab\repo\Executor.ps1`: Remote tool deployment; imports Executors, tries methods in chain (WinRM → SMB+WMI → WMI)
- `C:\DFIRLab\repo\QuickAiRLaunch.ps1`: Persistent job manager; WinForms UI, named mutex for singleton, job queue concurrency control
- `C:\DFIRLab\repo\Report.html`: Browser-based GUI; loaded from disk, fully offline, supports drag-and-drop JSON

**Core Infrastructure:**

- `C:\DFIRLab\repo\Modules\Core\Connection.psm1`: WinRM session creation, capability probes (PSVersion, HasCIM, HasNetTCPIP, HasDNSClient), hostname resolution

**Collection Plugins:**

- `C:\DFIRLab\repo\Modules\Collectors\Processes.psm1`: ~800 lines; process enumeration via CIM (PS 5.1+), WMI (PS 2.0+), .NET fallback; includes signature verification and SHA256 computation

**Remote Execution:**

- `C:\DFIRLab\repo\Modules\Executors\SMBWMI.psm1`: ~600 lines; SMB share enumeration, binary copy via .NET SMB client, execution via WMI Win32_Process.Create

**Launcher:**

## Naming Conventions

**Files:**

- `Collector.ps1`, `Executor.ps1`, `QuickAiRLaunch.ps1`: Entry points (PascalCase, single responsibility)
- `TestSuite.ps1`, `Register-QuickAiRProtocol.ps1`: Helper scripts (PascalCase, Verb-Noun pattern)
- `*.psm1`: PowerShell script modules (lowercase, underscore separator for multi-word)
  - `Connection.psm1`, `Output.psm1` (Core modules)
  - `Processes.psm1`, `Network.psm1` (Collectors)
  - `WinRM.psm1`, `SMBWMI.psm1` (Executors)
  - `JobQueue.psm1`, `PipeListener.psm1` (Launcher)
- `T##_*.ps1`: Test scripts (numbered T01-T10, underscore separator)
- `##_*.html`, `##_*.js`: Report components (numbered 01-10, underscore separator)

**Directories:**

- PascalCase: `Modules`, `Collectors`, `Executors`, `Core`, `Launcher`, `ReportStages`, `DFIROutput`, `ExecutionResults`
- All-caps acronyms: `DFIR` (Digital Forensics and Incident Response)

**PowerShell Functions:**

- Verb-Noun pattern: `Get-TargetCaps`, `Invoke-Collector`, `Invoke-Executor`, `Test-PrivateIP`, `Write-JsonOutput`
- Verb-Noun-Adjective for detailed: `Resolve-TargetHostname`, `ConvertFrom-TcpStateInt`
- Private helpers: `_sha256`, `_validateInput` (leading underscore inside modules)

**Variables:**

- Param variables: `$Target`, `$Credential`, `$OutputPath` (PascalCase)
- Function locals: `$isLocal`, `$session`, `$result` (camelCase)
- Script-scope: `$script:LogFile`, `$script:StatusOrder` (lowercase with script: prefix)
- Constants: `$MAX_RETRY`, `$COLLECTOR_VERSION` (UPPER_SNAKE_CASE)

## Where to Add New Code

**New Collector (e.g., registry hives, Windows Event Log):**

1. Create: `C:\DFIRLab\repo\Modules\Collectors\NewArtifact.psm1`
2. Export: `function Invoke-Collector { param(-Session, -TargetPSVersion, -TargetCapabilities) }` returning `@{ data = @(...); source = "cim|wmi|legacy"; errors = @() }`
3. Pattern: Import Core\DateTime.psm1 for timestamp formatting; check $TargetCapabilities before using CIM
4. Fallback chain: Try CIM (PS 5.1+) → fallback to WMI (PS 2.0+) → fallback to cmd/executable
5. Auto-discovered: Collector.ps1 finds all `Modules\Collectors\*.psm1` files alphabetically
6. Test: Add T## test script to validate new artifact in Tests/ directory

**New Executor Method (e.g., SSH for Linux targets, RDP with remote command):**

1. Create: `C:\DFIRLab\repo\Modules\Executors\MethodName.psm1`
2. Export: `function Invoke-Executor { param(-ComputerName, -Credential, -LocalBinaryPath, -RemoteDestPath, -Arguments, -AliveCheckSeconds, -BinaryTypeOverride) }` returning ordered hashtable with ExecutionId, ComputerName, Method, States array, FinalState
3. Pattern: Each state transition recorded with timestamp (UTC ISO8601) and detail message
4. Terminal states: 'ALIVE', 'SFX_LAUNCHED', 'CONNECTION_FAILED', 'TRANSFER_FAILED', 'LAUNCH_FAILED', 'FAILED', 'TIMEOUT'
5. Integration: Update Executor.ps1 to add method to the fallback chain (see lines 213-252)
6. Test: Add T## test script to Tests/ directory

**New Report Tab (e.g., Registry keys, Event Log analysis):**

1. Create: `C:\DFIRLab\repo\ReportStages\##_tabname.js` (use next available number)
2. Pattern: Export `window.Tab_TabName = { init: function(container) { ... }, render: function(data) { ... }, search: function(term) { ... } }`
3. Data access: Use `window.g_data.currentHost.artifact` (see 03_core.js for data structure)
4. UI pattern: Create table, support sorting/filtering, highlight matches for search
5. Integration:
   a. Update `02_shell.html` to add tab button
   b. Update `03_core.js` Tab[] array to register new tab
   c. Register new tab object in 03_core.js registerTabs() function
6. Build: Concatenate all ReportStages/* files in order into Report.html (see build_report.ps1)
7. Test: Add T## test script to Tests/ directory

**New Utility Module (e.g., shared validation, encryption):**

1. Create: `C:\DFIRLab\repo\Modules\Core\Utility.psm1`
2. Pattern: No external dependencies; functions start with descriptive Verb-Noun name
3. Exports: List all public functions in header block
4. Integration: Import in Collector.ps1 (line 56-59) or individual modules as needed
5. Test: Add test cases to existing T## files or create new one

**Configuration/Constants:**

- PowerShell scripts: Define at top after #Requires, inside #region --- Constants ---
- JavaScript: Add to 03_core.js at top of file inside // ---- Configuration section
- Example: `$MAX_RETRY = 3`, `$EXECUTOR_VERSION = "1.8"`
